# ТЕХНИЧЕСКАЯ СПЕЦИФИКАЦИЯ (SPEC.md)
## Проект: Electron Drive Sync (DJ Library) — Расширенная детерминированная версия

### 1. Архитектурный анализ и скрытые риски (Chain of Thought & Deep Dive)

Перед детализацией фаз осуществлена деконструкция критических узлов бизнес-логики, которые в базовых спецификациях обозначаются верхнеуровнево. Без их математического и алгоритмического уточнения распределенные агенты неизбежно столкнутся со структурными тупиками (Deadlocks) при параллельной разработке.

#### 1.1. Матрица состояний Diff Engine (Разрешение конфликтов)
Так как Google Drive API v3 использует метки времени `modifiedTime` в формате RFC 3339 (`YYYY-MM-DDTHH:mm:ss.sssZ`), а локальные файловые системы (особенно под управлением Windows/macOS) могут возвращать округленные значения или работать в миллисекундах `mtimeMs`, прямое сравнение дат "в лоб" приведет к ложным срабатываниям.

**Решение:** Внедрить временной буфер (Epsilon-значение в 2 секунды) при сравнении таймстемпов. Если хэши MD5 различаются, но разница во времени изменения меньше Epsilon, приоритет отдается логике предотвращения перезаписи пользовательских данных.

#### 1.2. Специфика "Плоской структуры" Google Drive
Запрос списка файлов (Итерация по `drive.files.list`) должен явно использовать параметр фильтрации `q` с инъекцией родительского ID папки, иначе API вернет файлы из корня или корзины.
* **Шаблон запроса:** `'<parent_id>' in parents and trashed = false`
* **Обязательные поля ответа (fields):** `nextPageToken, files(id, name, md5Checksum, modifiedTime, mimeType)`

### 2. Актуализированная декомпозиция фаз для агентов

#### ФАЗА 1: Архитектура и базовая настройка (Агент: Architect-Dev)
* **Логика и ограничения:** Построение изолированной среды Electron. Использование флагов `contextIsolation: true` и `nodeIntegration: false` является обязательным стандартом безопасности. `preload.js` выступает в роли строгого шлюза (API Bridge), экспортируя только безопасные и строго типизированные методы через `contextBridge.exposeInMainWorld`.
* **Входные данные (Inputs):** Файл `package.json` с зафиксированными версиями зависимостей: `"electron": "^31.0.0"`, `"electron-builder": "^24.13.3"`. Архитектурный регламент структуры каталогов.
* **Выходные результаты (Outputs):** Файл безопасного моста `src/preload/index.js`:
```javascript
const { contextBridge, ipcRenderer } = require('electron');

contextBridge.exposeInMainWorld('electronAPI', {
  selectFolder: () => ipcRenderer.invoke('dialog:select-folder'),
  loginGoogle: () => ipcRenderer.send('auth:login'),
  startSync: (config) => ipcRenderer.send('sync:start', config),
  onSyncProgress: (callback) => ipcRenderer.on('sync:progress', (e, v) => callback(v)),
  onAuthSuccess: (callback) => ipcRenderer.on('auth:success', (e, p) => callback(p))
});
```
* **Тестирование:** Автоматизированный тест Playwright/Electron проверяет, что `window.electronAPI` доступен в контексте рендерера, а `window.require` или `process` возвращают `undefined`.

#### ФАЗА 2: UI/UX Прототипирование (Агент: Stitch-Designer)
* **Логика и ограничения:** Создание реактивного интерфейса, способного плавно обрабатывать потоковые данные (до 100 обновлений UI в секунду при хэшировании множества мелких файлов) без утечек памяти. Массив логов на стороне UI должен иметь жесткий лимит (например, последние 200 строк).
* **Входные данные (Inputs):** Спецификация состояний интерфейса: `IDLE`, `AUTH_PENDING`, `AUTHORIZED`, `SYNC_HASHING`, `SYNC_PROCESSING`, `SYNC_COMPLETED`, `ERROR`.
* **Выходные результаты (Outputs):** Набор реактивных веб-компонентов. Компонент `SyncWidget`, принимающий строго типизированный объект прогресса:
```json
{
  "status": "SYNC_PROCESSING",
  "currentFile": "audio/techno/kick_drum_128bpm.mp3",
  "processedFiles": 42,
  "totalFiles": 150,
  "percentage": 28.0,
  "speed": "4.2 MB/s"
}
```
* **Тестирование:** Изолированный компонентный тест (Cypress/Storybook) на обработку краевых значений прогресса (0%, 100%) и длинных имен файлов с CSS-обрезкой через `text-overflow: ellipsis`.

#### ФАЗА 3: Интеграция Google Auth с PKCE (Агент: Auth-Specialist)
* **Логика и ограничения:** Реализация OAuth 2.0. Main Process инициализирует локальный HTTP Loopback сервер на случайном свободном порту (порт `0` в Node.js). Генерируется криптографический ключ `code_verifier` и его хэш `code_challenge` (механизм PKCE). Системный браузер открывается по внешней ссылке. После авторизации Google выполняет редирект на локальный сервер, который перехватывает авторизационный код, закрывает порт и обменивает код на токены.
* **Входные данные (Inputs):** `client_id` приложения из Google Cloud Console, запрашиваемые OAuth Scopes (`drive.file`, `profile`).
* **Выходные результаты (Outputs):** Модуль `auth.service.js`, безопасно сохраняющий `refresh_token` в защищенное системное хранилище ОС (через `safeStorage` в Electron):
```json
{
  "access_token": "ya29.a0Axoo...",
  "expiry_date": 1718912345678,
  "user": { "name": "Nikita Shilin", "picture": "https://..." }
}
```
* **Тестирование:** Мок-тестирование сервера авторизации Jest. Проверка обработки ошибок типа `access_denied` (отказ пользователя): приложение должно корректно закрыть локальный порт и передать событие ошибки в UI.

#### ФАЗА 4: Разработка движка синхронизации (Агент: Core-Logic-Dev)
* **Логика и ограничения:** Ядро системы, исполняемое строго в Main-процессе. Ограничение параллельных сетевых запросов регулируется через пул конкурентности (например, `p-limit`). Потоковое вычисление MD5 (через `crypto.createHash`) исключает загрузку файлов целиком в ОЗУ, предотвращая падение процесса по лимиту памяти heap.
* **Входные данные (Inputs):** Абсолютный путь `localPath`, валидный `accessToken`, конфигурация пула запросов (`maxConcurrentRequests = 3`).
* **Выходные результаты (Outputs):** Модуль Diff Engine, формирующий чистые массивы задач на синхронизацию:
```json
{
  "toUpload": [
    { "localPath": "/path/track1.mp3", "relativePath": "track1.mp3", "md5": "098f..." }
  ],
  "toDownload": [
    { "driveId": "1a2b3c...", "relativePath": "dubstep/bass.wav", "md5": "ad02..." }
  ]
}
```
* **Тестирование:** Интеграционный тест дифференциального модуля на фиктивном дереве файлов во временной директории `tmp`. Стресс-тест сети с эмуляцией ошибки `403 Usage Limits Exceeded` для проверки механизма Exponential Backoff.

#### ФАЗА 5: Интеграция и Связывание (Агент: Integrator)
* **Логика и ограничения:** Оркестрация жизненного цикла приложения. Связывание реактивных интерфейсных методов с вычислительным ядром, настройка централизованного перехвата глобальных исключений (`process.on('uncaughtException')`) и конфигурация параметров финального деплоя.
* **Входные данные (Inputs):** Интегрированные и протестированные модули Фаз 1-4.
* **Выходные результаты (Outputs):** Сконфигурированный файл сборки `electron-builder.json`, готовый к запуску пайплайна CI/CD:
```json
{
  "appId": "com.nikita.electron-drive-sync",
  "productName": "Electron Drive Sync",
  "directories": { "output": "dist" },
  "win": { "target": ["nsis"] },
  "mac": { "target": ["dmg"], "category": "public.app-category.utilities" }
}
```
* **Тестирование:** Сквозной E2E-тест (End-to-End) в симулированной среде: прохождение авторизации, изменение локального тестового файла, проверка успешного апдейта файла на Google Drive и корректного завершения процесса со статусом `SYNC_COMPLETED`.
