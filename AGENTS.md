# МАТРИЦА РАСПРЕДЕЛЕНИЯ АГЕНТОВ (AGENTS.md)
## Проект: Electron Drive Sync (DJ Library) — Расширенная детерминированная версия

Данный документ регламентирует изолированные контексты разработки для каждого задействованного агента. Изменение межпроцессных интерфейсов (IPC) или нарушение установленных правил без предварительного апдейта общего контракта (SPEC.md) строго запрещено.

---

### 1. Агент: Architect-Dev

* **Role (Зона ответственности):** Инициализация и базовая настройка изолированной среды Electron. Проектирование безопасного слоя межпроцессного взаимодействия (IPC Bridge) и организация архитектуры каталогов проекта. Контроль базовой безопасности приложения.
* **Context (Доступные разделы SPEC.md):**
    * Раздел 2: ФАЗА 1 (Архитектура и базовая настройка)
    * Раздел 3: QA Matrix (Тест QA-101)
* **Rules (Запреты и архитектурные ограничения):**
    * **Критически важно:** Настройки окон Electron обязаны строго содержать фреймворк безопасности: `contextIsolation: true` и `nodeIntegration: false`. Любое нарушение блокирует сборку.
    * Запрещен прямой доступ к модулям Node.js или API Electron из контекста рендерера (UI).
    * Файл `preload.js` должен являться единственным шлюзом (`contextBridge.exposeInMainWorld`), экспортирующим строго типизированные и безопасные методы.
    * Версии зависимостей в `package.json` должны быть жестко зафиксированы: `"electron": "^31.0.0"`, `"electron-builder": "^24.13.3"`.
* **Expected Output (Формат отдачи кода):**
    * Конфигурационный файл `package.json` с зафиксированными зависимостями.
    * Скрипт главной точки входа Main-процесса (`src/main/index.js`) с инициализацией безопасного окна.
    * Валидный файл безопасного моста `src/preload/index.js` с экспортом объекта `electronAPI`.
    * Автоматизированный тест на базе Playwright/Electron для проверки изоляции контекста (`window.require` должен возвращать `undefined`).

### 2. Агент: Stitch-Designer

* **Role (Зона ответственности):**
    Разработка реактивного пользовательского интерфейса, проектирование UI-компонентов и обеспечение стабильного FPS при обработке высокоинтенсивных потоков данных от движка синхронизации.
* **Context (Доступные разделы SPEC.md):**
    * Раздел 2: ФАЗА 2 (UI/UX Прототипирование)
    * Раздел 3: QA Matrix (Тест QA-201)
* **Rules (Запреты и архитектурные ограничения):**
    * Интерфейс обязан бесперебойно обрабатывать до 100 обновлений в секунду через IPC-канал без падения производительности и утечек памяти.
    * **Ограничение на логи:** Массив выводимых на экран логов синхронизации должен иметь жесткий лимит — не более последних 200 строк. Старые записи должны своевременно вытесняться из памяти.
    * Интерфейс должен строго поддерживать конечный автомат состояний: `IDLE`, `AUTH_PENDING`, `AUTHORIZED`, `SYNC_HASHING`, `SYNC_PROCESSING`, `SYNC_COMPLETED`, `ERROR`.
    * Все длинные имена файлов в интерфейсе должны иметь CSS-обрезку через свойство `text-overflow: ellipsis`.
* **Expected Output (Формат отдачи кода):**
    * Набор реактивных веб-компонентов интерфейса (HTML/CSS/JS).
    * Компонент `SyncWidget`, принимающий строго типизированный JSON-объект прогресса (`status`, `currentFile`, `processedFiles`, `totalFiles`, `percentage`, `speed`).
    * Изолированные компонентные тесты (Cypress/Storybook), эмулирующие граничные состояния прогресса (0%, 100%) и оверфлоу длинных строк.

### 3. Агент: Auth-Specialist

* **Role (Зона ответственности):**
    Реализация безопасного сквозного протокола авторизации Google OAuth 2.0 с использованием механизма PKCE и организация защищенного хранения сессии на целевой ОС.
* **Context (Доступные разделы SPEC.md):**
    * Раздел 2: ФАЗА 3 (Интеграция Google Auth с PKCE)
    * Раздел 3: QA Matrix (Тест QA-301)
* **Rules (Запреты и архитектурные ограничения):**
    * Запрещено использовать Implicit Flow или хардкодить секреты клиента. Обязателен механизм PKCE (`code_verifier` и криптографический хэш `code_challenge`).
    * Локальный HTTP Loopback сервер для перехвата редиректа должен запускаться исключительно в Main Process на случайном свободном порту (использовать порт `0` в Node.js).
    * **Безопасность токенов:** Запрещено сохранять `refresh_token` в localStorage, IndexedDB или незащищенный JSON-файл. Сохранение должно выполняться только через нативный API Electron — `safeStorage`.
    * В случае сетевых сбоев или отказа пользователя (`access_denied`) агент обязан корректно освободить локальный порт, закрыть сервер и передать ошибку в UI.
* **Expected Output (Формат отдачи кода):**
    * Самодостаточный модуль `src/main/services/auth.service.js`.
    * Конфигурация необходимых OAuth Scopes (`drive.file`, `profile`) и интеграция с системным браузером.
    * Сценарии мок-тестирования сервера авторизации на Jest, включая обработку обрыва соединения в момент редиректа.

### 4. Агент: Core-Logic-Dev

* **Role (Зона ответственности):**
    Разработка вычислительного ядра системы, движка дифференциального анализа (Diff Engine) и отказоустойчивого взаимодействия с Google Drive API v3.
* **Context (Доступные разделы SPEC.md):**
    * Раздел 1: Архитектурный анализ и скрытые риски (пп. 1.1 и 1.2)
    * Раздел 2: ФАЗА 4 (Разработка движка синхронизации)
    * Раздел 3: QA Matrix (Тест QA-401)
* **Rules (Запреты и архитектурные ограничения):**
    * **Изоляция выполнения:** Весь код движка исполняется строго в Main-процессе.
    * Запрещено читать файлы целиком в ОЗУ для вычисления контрольных сумм. Использовать исключительно потоковое вычисление MD5 через `crypto.createHash`.
    * Количество параллельных сетевых запросов к API должно быть жестко ограничено пулом конкурентности (например, через `p-limit` со значением `maxConcurrentRequests = 3`).
    * **Алгоритм сравнения дат (Diff Engine):** Прямое сравнение таймстемпов запрещено. Необходимо закладывать временной буфер (Epsilon-значение в 2 секунды) для компенсации разницы между RFC 3339 (Google Drive) и `mtimeMs` (локальная ФС). При несовпадении MD5 и дельте времени меньше Epsilon — отдавать приоритет защите пользовательских данных от перезаписи.
    * **Специфика Google Drive:** Запрос списков файлов должен обязательно содержать фильтрацию по родителю (`q: '<parent_id>' in parents and trashed = false`) и точечный ограничитель полей (`fields: 'nextPageToken, files(id, name, md5Checksum, modifiedTime, mimeType)'`).
    * **Регистрозависимость:** Движок должен обрабатывать нечувствительность к регистру символов (Case-insensitivity) файловой системы Windows во избежание бесконечных циклов перезаписи файлов с именами в разном регистре.
    * Обязательна реализация алгоритма Exponential Backoff при получении ошибок сетевых лимитов (`403 Usage Limits Exceeded`).
* **Expected Output (Формат отдачи кода):**
    * Модуль Diff Engine (`src/main/engine/diff.js`), возвращающий чистые структуры массивов `toUpload` и `toDownload`.
    * Интеграционные тесты дифференциального анализа на фиктивном дереве файлов внутри директории `tmp`.
    * Стресс-тесты сетевого слоя с эмуляцией задержек и ошибок 403.

* **Role (Зона ответственности):**
    Глобальная оркестрация жизненного цикла приложения, связывание разработанных модулей через IPC-каналы, организация централизованного логирования, перехвата критических ошибок и конфигурация финальной дистрибуции.
* **Context (Доступные разделы SPEC.md):**
    * Раздел 2: ФАЗА 5 (Интеграция и Связывание)
    * Раздел 3: QA Matrix (Тест QA-501)
    * Сквозной анализ всех связанных интерфейсов спецификации.
* **Rules (Запреты и architectural ограничения):**
    * Запрещено произвольно менять названия IPC-каналов, определенных на Фазе 1 и Фазе 2.
    * Обязательно наличие глобального перехватчика необработанных исключений (`process.on('uncaughtException')`), предотвращающего «тихое» падение приложения без уведомления пользователя.
    * Конфигурация сборщика должна производить нативные пакеты строго под целевые ОС: Windows (`nsis`) и macOS (`dmg`), учитывая нативные бинарные зависимости ядра (такие как `safeStorage`).
* **Expected Output (Формат отдачи кода):**
    * Связующий код инициализации приложения в главном файле, оркеструющий вызовы `Auth-Specialist` и `Core-Logic-Dev` и транслирующий прогресс в UI-компоненты `Stitch-Designer`.
    * Полный конфигурационный файл сборки `electron-builder.json` со всеми прописанными метаданными (`appId`, `productName`, `targets`).
    * Сценарии сквозного E2E-тестирования (End-to-End), проверяющие полную цепочку: Авторизация -> Изменение файла -> Проверка апдейта на Google Drive -> Статус `SYNC_COMPLETED`.
